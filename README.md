Лабораторна робота №1

Я використав теги HTML5 для створення чіткої ієрархії документа: header для заголовка, main для основного контенту та section для логічного поділу форми й таблиці/списку.

Таблиця реалізована за всіма правилами семантики: з використанням блоків thead (для заголовків), tbody (для динамічних даних) та тегів th, що забезпечує коректну структуру даних.

Кожному полю введення (input, select) я призначив унікальний id та зв’язав його з відповідним елементом label через атрибут for.

Згідно з Варіантом №1, я створив форму для заявок на доступ, що містить: текстовий інпут для імені, поле вибору дати (type="date") та випадаючий список (select) для типу доступу (Студент, Викладач, Гість).

Кнопки форми мають чітко визначені типи: type="submit" для ініціації валідації та додавання запису, та type="button" для допоміжних дій (видалення записів), що запобігає випадковому перезавантаженню сторінки.

Для побудови інтерфейсу я використав Flexbox. Головний контейнер має display: flex, що дозволило мені реалізувати зручну структуру (наприклад, форма з одного боку, список з іншого, або акуратне розміщення полів).

Я встановив глобальне правило box-sizing: border-box, щоб внутрішні відступи (padding) та межі (border) не змінювали розрахункову ширину елементів, забезпечуючи стабільність сітки.

Реалізовано клієнтську валідацію: якщо поле невалідне, за допомогою JS на нього вішається відповідний CSS-клас (який підсвічує поле червоним) і адресно виводиться повідомлення про помилку під конкретним інпутом. JavaScript код чітко структуровано з виділенням логічних блоків (читання форми, валідація, рендер).



Лабораторна робота №2

Я перевів проєкт на TypeScript, налаштувавши tsconfig.json та збірку через tsx для режиму розробки. Це дозволило мені уникнути використання any та впровадити сувору типізацію для всіх DTO та бізнес-сутностей.

Розділив код на логічні шари згідно з принципами Clean Architecture: Routes/Controllers (прийом запитів), Services (бізнес-логіка), Repositories (in-memory доступ до даних) та DTOs (опис структур запитів та відповідей).

Впровадив стандартизацію контрактів: усі відповіді для списків мають уніфікований формат { data: [...] }, а помилки повертаються в єдиному форматі з кодом та деталями.

Реалізував повний набір CRUD-маршрутів для сутностей Варіанта №1: Користувачі (Users) та Заявки на доступ (AccessRequests).

Я дотримався REST-стандартів: використання множини в URL (/api/requests) та відповідних HTTP-методів: GET (читання), POST (201 Created), PATCH (200 OK) та DELETE (204 No Content).

Для ідентифікації об'єктів реалізував генерацію унікальних ID на бекенді, що гарантує цілісність даних у пам'яті.

Розробив систему валідації вхідних даних (DTO): перевірка обов'язкових полів, відповідність типів та контроль довжини рядків.

Мною реалізована централізована обробка помилок через middleware. Будь-яка помилка (400 Bad Request, 404 Not Found або 500 Internal Server Error) перехоплюється та повертається клієнту у стабільному JSON-форматі.

Додала логування кожного запиту: у консоль виводиться метод, шлях, код відповіді та час виконання запиту в мілісекундах.

Реалізував можливість фільтрації списку заявок через query-параметри (GET /api/requests?status=Pending&limit=10).



Лабораторна робота №3 (SQLite)

Як запустити проєкт
Встановити залежності: npm install

Запустити сервер у режимі розробки: npm run dev

Для наповнення бази тестовими даними (Seed): npm run seed

Сервер автоматично підніметься на http://localhost:3000

Робота з базою даних
Файл бази даних створюється автоматично у папці ./data під ім'ям app.db.

Папка ./data створюється при першому запуску, якщо її ще немає.

Я реалізував власну систему міграцій: при запуску сервера автоматично читається папка src/migrations/, перевіряється таблиця schema_migrations, і виконуються лише нові SQL-файли.

База даних містить три доменні таблиці та таблицю для міграцій:

Таблиця Users (Користувачі):

id: INTEGER PRIMARY KEY

email: TEXT NOT NULL UNIQUE

name: TEXT NOT NULL

createdAt: TEXT NOT NULL

Таблиця AccessRequests (Заявки на доступ):

id: INTEGER PRIMARY KEY

userId: INTEGER NOT NULL

date: TEXT NOT NULL

accessType: TEXT NOT NULL

comments: TEXT

status: TEXT NOT NULL DEFAULT 'Pending' CHECK (status IN ('Pending', 'Approved', 'Rejected'))

createdAt: TEXT NOT NULL

FOREIGN KEY(userId) REFERENCES Users(id) ON DELETE CASCADE

Таблиця Approvals (Погодження заявок):

id: INTEGER PRIMARY KEY

requestId: INTEGER NOT NULL UNIQUE

approverId: INTEGER NOT NULL

decision: TEXT NOT NULL CHECK (decision IN ('Approved', 'Rejected'))

reason: TEXT

createdAt: TEXT NOT NULL

FOREIGN KEY(requestId) REFERENCES AccessRequests(id) ON DELETE CASCADE

FOREIGN KEY(approverId) REFERENCES Users(id) ON DELETE RESTRICT

Приклади запитів (API Endpoints)
Отримання всіх заявок з лімітом: GET /api/requests?limit=5

Повний список із JOIN (Заявка + Автор + Рішення): GET /api/requests/detailed

Агрегація даних (Статистика за статусами): GET /api/requests/stats

Створення заявки: POST /api/requests
Тіло: {"userId": 1, "date": "2026-06-10", "accessType": "Student", "comments": "Потрібен доступ для ЛР"}

Комплексна операція погодження (Транзакція на 2 таблиці): POST /api/requests/1/approve
Тіло: {"approverId": 2, "decision": "Approved", "reason": "Дозволено"}

Демонстрація SQL Injection
У проєкті спеціально залишено ендпойнт /api/requests/search, який використовує пряму конкатенацію рядка в SQL-запит (у функції searchRequestsDangerous).

Приклад ін'єкції: http://localhost:3000/api/requests/search?q=test' OR '1'='1

Як це працює: Запит перетвориться на SELECT * FROM AccessRequests WHERE comments LIKE '%test' OR '1'='1%'. Оскільки умова '1'='1' завжди істинна, сервер поверне абсолютно всі записи з таблиці, ігноруючи реальний фільтр. Це демонструє критичну небезпеку "сирих" запитів без використання параметризації.
